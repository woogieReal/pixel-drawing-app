## 📊 `canvases` 테이블 상세 가이드

이 테이블은 픽셀 하나하나를 개별 행(Row)으로 저장하는 대신, **하나의 도화지(Canvas) 전체를 하나의 행**에 담는 방식입니다.

### 1. 컬럼별 역할 분담

| 컬럼명 | 데이터 타입 | 역할 및 비즈니스 로직 |
| --- | --- | --- |
| **`canvas_id`** | `SERIAL` | 각 캔버스의 고유 식별자입니다. API 호출 시(`GET /canvas/1`) 기준으로 사용됩니다. |
| **`width`** | `INTEGER` | 캔버스의 가로 픽셀 수입니다. (예: 64, 128) **좌표 계산의 기준값**이 됩니다. |
| **`height`** | `INTEGER` | 캔버스의 세로 픽셀 수입니다. 캔버스의 전체 크기를 결정합니다. |
| **`pixel_data`** | **`BYTEA`** | **핵심 데이터**입니다. 모든 픽셀의 RGB 값을 바이너리(Buffer) 형태로 이어 붙인 덩어리입니다. |
| **`updated_at`** | `TIMESTAMP` | 마지막으로 픽셀이 수정된 시간입니다. 캐시 갱신 여부를 판단할 때 유용합니다. |

---

### 2. `pixel_data` (BYTEA)의 내부 구조

`BYTEA` 타입은 Node.js의 `Buffer` 객체와 1:1로 매칭됩니다. 픽셀 하나를 **RGB(3바이트)**로 표현한다면, 전체 데이터 크기는 다음과 같이 결정됩니다.

> **전체 크기(Bytes)** = $width \times height \times 3$
> * 64x64 캔버스: $4,096 \times 3 = 12,288$ Bytes (약 12KB)
> * 128x128 캔버스: $16,384 \times 3 = 49,152$ Bytes (약 48KB)
> 
>

### 3. 백엔드 로직 구현을 위한 핵심 공식

사용자가 특정 좌표 $(x, y)$에 색상(#FF5733)을 칠하겠다고 API를 쏘면, 백엔드는 전체 바이너리 뭉치에서 **어느 지점을 수정할지** 계산해야 합니다.

#### 💡 오프셋(Offset) 계산식

하나의 픽셀이 3바이트(R, G, B)를 차지하므로, $(x, y)$ 위치의 시작점은 다음과 같습니다.


$$\text{Offset} = (y \times \text{width} + x) \times 3$$

* **Red 값 위치:** `offset`
* **Green 값 위치:** `offset + 1`
* **Blue 값 위치:** `offset + 2`

---

### 4. 구현 시 주의사항 (대량 트래픽 관점)

1. **초기화 (Initialization):**
캔버스를 처음 생성할 때, `pixel_data`는 모두 흰색(`0xFF`)으로 채워진 정해진 크기의 Buffer로 미리 생성해서 채워 넣어야 합니다.
2. **부분 업데이트의 한계:**
PostgreSQL의 `BYTEA`는 특정 부분만 수정하는 것이 불가능합니다. 전체를 읽어서 수정 후 다시 전체를 써야 합니다.
* **해결책:** 트래픽이 몰릴 때는 **Redis**에서 이 작업을 수행하고, DB에는 일정 주기마다 **스냅샷**을 찍듯 저장하는 것이 좋습니다.


3. **조회 성능:**
클라이언트는 JSON 형태의 좌표 배열보다, 이 바이너리 데이터를 통째로 받아가서 브라우저 `Canvas API`로 그리는 것이 훨씬 빠릅니다.
